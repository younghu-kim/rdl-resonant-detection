### 🐛 핵심 발견 2: κ_nn 정의 불일치

**스크립트 108행**:
```python
kappa_nn = 1.0 / (d_n**2) + 1.0 / (d_np1**2)  # = 8/gap²
```

**문제**: 이것은 Hadamard 전개의 실제 기여가 아님.

설계자가 정확히 진단한 바와 같이, Hadamard 전개에서 s = 1/2 + im일 때:
```
1/(s - ρₙ) = 1/(i(m - γₙ)) = -i/(m - γₙ)
1/(s - ρₙ₊₁) = 1/(i(m - γₙ₊₁)) = -i/(m - γₙ₊₁)
```
중간점에서 m - γₙ = +gap/2, m - γₙ₊₁ = -gap/2 이므로:
- 두 항의 합 = -i/(gap/2) + i/(gap/2) = **0** (완전 상쇄)

따라서:
- **올바른 κ_nn** = |두 항의 합|² = 0
- **스크립트의 κ_nn** = |첫 항|² + |둘째 항|² = 8/gap² (교차항 무시)
- κ_nn/κ_total 평균 = 3.672 > 1: 스크립트의 κ_nn이 실제 기여보다 항상 과대 → **정의 자체가 부적합**

## 설계자 피드백

1. **bundle_utils.py h=10^{-20} 문제**: 이 h는 t<100 구간에서만 적절. t>100에서는 ξ의 절대값이 지수 성장하여 차분 정밀도가 소진됨. **향후 모든 스크립트에서 t>100 구간을 다룰 때 dps≥100 또는 h 조정 필요.** 이것은 알려진 함정에 추가해야 함.

2. **cap 원인 진단 부정확**: 설계자는 "중간점 취소 때문에 cap 발생"이라 진단했으나, 실제로는 수치미분 h의 유효자릿수 소진이 주 원인. 중간점에서 nn 기여가 상쇄되는 것은 사실이나, 이것이 |ξ'/ξ|를 0으로 만드는 것은 아님 (나머지 영점들의 기여가 있으므로). cap의 직접 원인은 `connection_zeta`가 쓰레기값을 반환하는 것.

3. **결과 파일 내 "⚠️ Control 실패 → 스크립트 버그 가능성 점검 필요"는 정확한 진단**: 실제로 스크립트 인프라 (bundle_utils.py의 h 고정)에 의한 문제.

### 알려진 함정 추가

- **bundle_utils.py h=10^{-20}**: t>100에서 수치미분 유효자릿수 소진. dps ≥ 2×20 + log₁₀|ξ(s)| 필요. 또는 h를 dps에 연동.
- **중간점 Hadamard 취소**: s = 1/2 + im에서 (γₙ, γₙ₊₁)의 중간점이면 두 최근접 영점의 L(s) 기여가 정확히 상쇄. κ_nn 정의 시 교차항(상쇄) 반영 필수.
- **κ_nn = 1/d² 합 vs |Σ 1/(s-ρ)|²**: 전자는 각 항의 절대값² 합, 후자는 벡터합의 절대값². 항상 전자 ≥ 후자 (삼각 부등식). 중간점에서는 전자 >> 후자.
