# 3단 연구 에이전트 시스템 — 설계 문서 v2

## 문제 진단 (2026-04-13)

### 반복되는 실패 패턴
1. **API 불일치**: 새 스크립트마다 같은 함수 시그니처 실수 반복
2. **수치 해상도**: 모노드로미 eps 방식, dps 부족 등 
3. **영점 탐색 실패**: 클로저/self 캡처 문제로 0개 반환
4. **결과 검증 누락**: "영점 0개"를 정상으로 받아들임

### 근본 원인
- **코드 재사용 부재**: 검증된 함수를 매번 다시 구현
- **검증 게이트 부재**: 결과가 비정상이어도 다음 단계로 진행
- **에이전트 간 맥락 단절**: 한 에이전트의 경험이 다음에 전달 안 됨

## 해결 구조

### Layer 0: 공용 인프라 (사람이 관리)
```
scripts/bundle_utils.py     — 검증된 다발/디리클레 함수
scripts/math_constitution.md — 12조 원칙
scripts/memory/semantic/     — 확립된 발견, 안티패턴
```
**원칙**: bundle_utils에 없는 함수는 직접 구현 금지. 필요하면 먼저 추가.

### Layer 1: 3단 에이전트 (Claude CLI로 실행)

```
Stage 1: 수학자 (판단) — Opus 고정
  - 읽기: 결과 파일, 보드, 논문
  - 쓰기: board/mathematician.md (지시 + 모델 지정)
  - 역할: 무엇을 할지, 왜 할지, 어떤 모델로 할지

Stage 2: 설계자 (실행) — 수학자 지정 (기본 Sonnet, 실패 시 Opus 자동 업그레이드)
  - 읽기: 수학자 지시, 체크리스트, bundle_utils
  - 쓰기: 스크립트 파일, board/executor.md (보고)
  - 역할: 어떻게 할지, 실행

Stage 3: 검토자 (통합) — Opus 고정
  - 읽기: 결과 파일, 모든 보드
  - 쓰기: 논문 TeX, board/reviewer.md, git
  - 역할: 맞는지 확인, 논문 반영, PDF 배포

보드 파일:
  board/mathematician.md  — 수학자 지시 (활성)
  board/executor.md       — 설계자 보고 (활성)
  board/reviewer.md       — 검토자 검증 (활성)
  board/archive/          — 구 4에이전트 보드 (비활성, 참조용)
```

### Layer 2: 자동 검증 게이트

각 Stage 사이에 자동 검증:

```
Stage 1 → [검증 1] → Stage 2
  - 수학자 보드에 "다음 작업" 있는가?
  - 이미 실행 중인 실험 없는가?

Stage 2 → [검증 2] → Stage 3  
  - 스크립트 문법 에러 없는가? (py_compile)
  - 결과 파일 존재하는가?
  - 결과에 "0개", NaN, 빈 값 없는가?

Stage 3 → [검증 3] → 완료
  - TeX 컴파일 성공?
  - git diff가 합리적인가? (100줄 이하?)
```

### Layer 3: 실패 복구

```
스크립트 에러 → Stage 2 재시도 (최대 3회)
결과 비정상 → Stage 1에 보고 (board에 경고 기록)
TeX 컴파일 실패 → Stage 3 재시도
3회 모두 실패 → 반성문 작성 + 사이클 중단
```

## 실행 방식

### 방식 A: 순차 CLI (현재)
```bash
./run_cycle.sh        # 3 × claude -p 호출
```
- 장점: 단순, 각 에이전트 맥락 독립
- 단점: 단일 턴, 멀티턴 디버깅 불가
- 용도: 판단 + 스크립트 작성까지

### 방식 B: 대화 내 Agent 도구
```
사람이 Claude Code 세션에서:
  Agent(stage1) → 결과 확인 → Agent(stage2) → 결과 확인 → Agent(stage3)
```
- 장점: 멀티턴 가능, 사람이 개입 가능
- 단점: 수동 개입 필요
- 용도: 복잡한 실험, 디버깅 필요 시

### 방식 C: nohup 장기 실행 (실험용)
```bash
nohup claude -p "..." > /tmp/stage2.log 2>&1 &
```
- 실험 실행은 스크립트 내에서 subprocess로 처리
- 장점: 무인 실행
- 단점: 실패 시 복구 불가

### 권장: A + B 혼합
- 일상: A로 자동 사이클
- 문제 발생: B로 사람 개입
- 장기 실험: 스크립트만 A로 생성, 실행은 nohup으로 별도

## 비용 최적화

| 단계 | 모델 | 이유 |
|------|------|------|
| Stage 1 (수학자) | Opus (고정) | 판단력 필요 |
| Stage 2 (설계자) | **수학자 지정** (기본 Sonnet) | 작업 난이도에 따라 동적 선택 |
| Stage 3 (검토자) | Opus (고정) | 비판적 검토 필요 |

→ 수학자가 `**모델**: sonnet/opus`로 지정. 미지정 시 Sonnet 기본.
→ Sonnet 실패 시 자동 Opus 업그레이드 재시도.
→ 단순 파라미터 변경 = Sonnet, 새 알고리즘/디버깅 = Opus

## 메트릭

시스템 효과 측정:
1. **API 에러율**: 스크립트 당 첫 실행 성공률 (목표: >80%)
2. **결과 비정상 검출률**: "영점 0개" 같은 비정상을 잡는 비율
3. **사이클 완료율**: 3단계 모두 정상 완료하는 비율
4. **사이클 당 시간**: 목표 < 15분
